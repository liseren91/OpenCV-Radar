# PRD — OpenCV Radar (рабочее название)

**Опенсорсный, бесплатный сервис: глубокий профиль кандидата → ежедневный авто-радар вакансий → тейлоринг CV под вакансию.**
BYO-ключ (пользователь приносит свой OpenAI/Anthropic-ключ). Vanilla HTML/JS, без сборки. Хостинг — GitHub Pages. Ежедневное обновление — GitHub Actions. Для разработчиков — self-host со своими скраперами.

Версия документа: 1.0 · Автор: Елизавета · Дата: 10.06.2026

---

## 1. Проблема и контекст

Современное CV — это одна страница. Но если человек работает 5+ лет, одна страница физически не вмещает его экспертизу: проекты, метрики, домены, инструменты, контекст решений. Из-за этого:

- кандидат сам не помнит и не формулирует половину своих сильных сторон;
- автоматический матчинг по «голому» CV промахивается — он видит 20% профиля;
- тейлоринг резюме под вакансию делается с нуля каждый раз и вручную.

При этом существующие AI-инструменты (resume-lm, Resume Matcher, разные SaaS-билдеры) решают только кусок — «перепиши CV под JD» — и почти никто не решает **извлечение скрытой экспертизы** и **персональный ежедневный радар** в одном месте, да ещё и бесплатно.

**Гипотеза ценности:** если один раз вытащить из человека «расширенный мастер-профиль» (богаче CV в разы) через адаптивный до-опрос, то и матчинг вакансий, и тейлоринг резко улучшаются — потому что у системы появляется полная картина, а не одна страница.

## 2. Принципы продукта (ограничения, которые нельзя нарушать)

1. **Бесплатно для пользователя и для автора.** Автор не финансирует ни LLM, ни инфраструктуру под каждого юзера.
2. **BYO-ключ.** Дорогую AI-работу оплачивает ключ самого пользователя. Ключ живёт только в браузере пользователя и никогда не уходит на сервер автора.
3. **Приватность по умолчанию.** CV, ответы до-опроса, история — в браузере (localStorage/IndexedDB). Сервер автора не хранит персональные данные.
4. **Легальность.** Публичный хостинг использует только официальные/открытые job-API. Скрапинг (LinkedIn и т.п.) — только в self-host, под ответственность пользователя.
5. **Низкий порог входа в код.** Vanilla HTML/JS, без сборки и без node_modules в основном фронтенде — чтобы контрибьютить могли и джуны, и продакты.
6. **Опенсорс-first.** Чистая лицензия, понятная структура, лёгкий первый PR.

## 3. Целевая аудитория

- **«Друзья» (нетехнари).** Открывают готовый сайт, проходят онбординг, вставляют свой LLM-ключ, пользуются. Ноль установки.
- **Разработчики/энтузиасты.** Клонируют репо, поднимают локально (Docker), подключают self-hosted скраперы, форкают, улучшают, шлют PR.
- **Сам автор** как первый и эталонный пользователь (профиль PM/MarTech/AI, гео Remote/Belgrade/EU).

## 4. Архитектура: ключевая идея

> **Общий пул вакансий — один на всех. Персонализация — у каждого в браузере.**

- Раз в день GitHub Action собирает «сырой» пул свежих вакансий из бесплатных API → `data/jobs.json`. Дёшево, делается единожды для всех.
- Браузер пользователя берёт `jobs.json` + его мастер-профиль (из localStorage) и **его** LLM-ключом ранжирует/матчит/переписывает. Платит ключ пользователя, прямо в браузере.

Так автор не платит ни за вакансии (free-тиры + один общий запрос), ни за LLM (BYO каждого).

### 4.1. Две «двери» из одного репозитория

| | Дверь A — Хостинг (друзья) | Дверь B — Self-host (разработчики) |
|---|---|---|
| Где | `<login>.github.io/opencv-radar` | `docker compose up` локально |
| Источник вакансий | бесплатные API (Adzuna, Remotive, hh.ru API, ATS-фиды) | то же + self-hosted скраперы (JobSpy: LinkedIn/Indeed/Glassdoor) |
| Обновление | GitHub Actions (общий `jobs.json`) | свой `cron` в воркере |
| Доставка свежего | «зашёл на сайт — увидел» | свой cron + опционально уведомления |
| LLM | BYO-ключ в браузере | BYO-ключ (или ключ в `.env`) |
| Скраперы | НЕТ (юридическая чистота хостинга) | ДА, под ответственность пользователя |

Один и тот же фронтенд обслуживает обе двери. Разница только в **источнике `jobs.json`**: на хостинге его пишет Action, в self-host — локальный воркер.

### 4.2. Поток данных (текстовая диаграмма)

```
[Дверь A — Хостинг]
GitHub Actions (cron, ежедневно)
   └─ scripts/fetch-jobs.mjs  ──ходит в── Adzuna / Remotive / hh.ru API / ATS-фиды
        └─ пишет data/jobs.json ──commit──▶ репо
                                              │
GitHub Pages (статика) ◀───────────────────── отдаёт index.html + jobs.json
   │
   ▼
Браузер пользователя
   ├─ CV (pdf.js/mammoth) → парсинг
   ├─ Глубокий до-опрос (LLM, ключ юзера) → «мастер-профиль» (localStorage)
   ├─ Матчинг jobs.json × мастер-профиль (LLM, ключ юзера) → ранжированный дашборд
   └─ Тейлоринг CV + cover letter + пробив вилки (LLM, ключ юзера)

[Дверь B — Self-host] то же, но jobs.json пишет локальный воркер,
   к которому подключаются скраперы (JobSpy) и собственный cron.
```

## 5. Ключевые user stories

**Онбординг и профиль**
- Как пользователь, я загружаю своё CV (PDF/DOCX) и вижу, что система корректно распознала структуру (опыт, навыки, проекты).
- Как пользователь, я прохожу адаптивный до-опрос, который задаёт уточняющие вопросы по тем местам, где CV «тонкий», и не задаёт лишнего по тому, что уже ясно.
- Как пользователь, я получаю «мастер-профиль» — расширенную структуру, которую могу отредактировать, экспортировать в JSON и импортировать на другом устройстве.

**Ключ и приватность**
- Как пользователь, я вставляю свой OpenAI/Anthropic-ключ один раз; он сохраняется только в моём браузере; я вижу явное уведомление об этом и кнопку «удалить ключ».
- Как пользователь, я могу выбрать провайдера (OpenAI или Anthropic) и модель.

**Дашборд вакансий**
- Как пользователь, я открываю дашборд и вижу свежие вакансии, отранжированные под мой мастер-профиль, с пояснением «почему подходит».
- Как пользователь, я фильтрую по гео (Remote/Belgrade/EU), грейду, тегам (AI/MarTech/R&D).
- Как пользователь, я вижу, что список обновляется ежедневно, и помечен значком «NEW».

**Тейлоринг**
- Как пользователь, я выбираю вакансию (или вставляю любую JD) и получаю CV, переписанное под неё, + cover letter.
- Как пользователь, я вижу оценку зарплатной вилки по рынку из легальных источников и сравнение с моими ожиданиями.

**Разработчик**
- Как разработчик, я поднимаю проект локально одной командой и добавляю новый источник вакансий, реализовав один адаптер по интерфейсу.
- Как разработчик, я включаю self-hosted скрапер и свой cron, не трогая фронтенд.

## 6. Технический стек

**Фронтенд (обе двери)**
- Чистый HTML + CSS + JavaScript (ES-модули), **без сборщика**. Подключение библиотек через CDN/`<script type="module">`.
- Парсинг CV: `pdf.js` (PDF), `mammoth.js` (DOCX) — обе работают в браузере.
- Хранение: `localStorage` (профиль, настройки, ключ), `IndexedDB` (файлы CV, история откликов — крупные данные).
- LLM-слой: тонкая абстракция `providers/` с адаптерами `openai.js` и `anthropic.js` (единый интерфейс `chat(messages, opts)`). Браузерные вызовы напрямую: OpenAI и Anthropic отдают CORS; для Anthropic добавляем заголовок `anthropic-dangerous-direct-browser-access: true`.
- UI можно оставить полностью на ванили; если захочется реактивности без сборки — опционально Alpine.js или Preact через CDN (не обязательно для MVP).

**Сбор вакансий (Дверь A, в GitHub Actions)**
- Скрипт на Node (`fetch-jobs.mjs`) — без зависимостей или с минимумом. Каждый источник = отдельный адаптер в `scripts/sources/`.
- Бесплатные источники для старта: **Adzuna API** (есть зарплатные данные), **Remotive**, **RemoteOK**, **Arbeitnow**, **Jooble API**, **hh.ru API** (`api.hh.ru` — официальный, для Сербии/EU полезен), публичные **ATS-фиды** (Greenhouse `boards-api`, Lever `postings`, Workable, Ashby) — отдают вакансии конкретных компаний легально и без ключа.
- Нормализация всех источников в единую схему вакансии (см. §7).

**Self-host (Дверь B)**
- `docker-compose.yml`: контейнер со статикой (любой статический сервер) + воркер.
- Воркер (Node или Python) запускает те же адаптеры **плюс** скраперы. Скрапинг — через **JobSpy** (Python-библиотека: LinkedIn/Indeed/Glassdoor/Google). Свой `cron` внутри контейнера.
- Скраперы изолированы в `selfhost/` и НЕ попадают в сборку GitHub Pages.

**Инфраструктура**
- **GitHub Pages** — хостинг статики (бесплатно).
- **GitHub Actions** — два workflow: `update-jobs.yml` (ежедневный cron, пишет `jobs.json`), `deploy-pages.yml` (деплой статики). Для публичных репозиториев минуты Actions бесплатны без лимита.
- Секреты job-API (например, Adzuna `app_id`/`app_key`) — в Secrets репозитория; используются только внутри Action, наружу уходит лишь публичный `jobs.json`.

## 7. Модель данных

**Единая схема вакансии (`jobs.json` — массив):**
```json
{
  "id": "stable-hash",
  "title": "Senior Product Manager, AI",
  "company": "Hostaway",
  "location": "Remote (Europe)",
  "remote": true,
  "url": "https://...",
  "source": "remotive",
  "posted_at": "2026-06-09",
  "salary": { "min": 43200, "max": 75900, "currency": "EUR", "source": "adzuna" },
  "description": "raw text...",
  "tags": ["AI", "Remote", "Senior"]
}
```

**Мастер-профиль (localStorage, редактируемый, экспортируемый JSON):**
```json
{
  "basics": { "name": "...", "english": "C1", "locations": ["Remote","Belgrade","EU"] },
  "headline_roles": ["Lead/Group PM","Head of Product","Product Owner"],
  "experience": [{ "company": "...", "role": "...", "achievements": ["метрика..."], "domains": ["MarTech"], "tools": ["..."] }],
  "hidden_expertise": ["...вытащенное до-опросом..."],
  "preferences": { "focus": ["AI","R&D","MarTech"], "exclude": ["US-only","sales","junior"] },
  "salary_expectation": { "min": 0, "currency": "EUR" }
}
```

Парсинг исходного CV → черновик профиля → до-опрос дополняет `hidden_expertise` и недостающие поля.

## 8. Глубокий до-опрос — сердце продукта (детальнее)

Это главный дифференциатор, поэтому в MVP качество вопросов важнее количества источников вакансий.

- LLM получает распарсенное CV и ищет «тонкие места»: роли без метрик, домены без деталей, периоды без проектов, навыки без подтверждения.
- Генерирует **адаптивные** вопросы пачками по 2–3, а не анкету на 40 пунктов. После каждого ответа решает, копать ли глубже или перейти дальше.
- Останавливается, когда профиль «насыщен» (новые вопросы перестают давать прирост информации) — показывает прогресс-бар «полнота профиля».
- Результат складывается в `hidden_expertise` и обогащает соответствующие записи опыта.
- Промпты до-опроса лежат в `prompts/` как редактируемые шаблоны (чтобы контрибьюторы улучшали их без кода).

Анти-цель: не превратиться в «ещё один скучный CV-билдер». Метрика качества — насколько профиль после опроса полнее исходного CV и насколько улучшается матчинг.

## 10. Scope и приоритеты

**Must have (MVP)**
- Загрузка и парсинг CV (PDF/DOCX) в браузере.
- BYO-ключ (OpenAI + Anthropic), хранение в браузере, явное уведомление о приватности, кнопка удаления.
- Глубокий адаптивный до-опрос → мастер-профиль (с экспортом/импортом JSON).
- `jobs.json` от 2–3 бесплатных источников (старт: Remotive + Adzuna + hh.ru API).
- Дашборд с клиентским матчингом под профиль + базовые фильтры (гео/грейд/теги).
- GitHub Actions ежедневный cron + деплой на Pages.
- README + CONTRIBUTING + лицензия.

**Should have**
- Тейлоринг CV под выбранную вакансию + cover letter.
- Пробив зарплатной вилки из легальных источников (Adzuna salary).
- Дедуп вакансий и пометка «NEW» (как в текущем дашборде).
- ATS-фиды (Greenhouse/Lever/Workable) как дополнительные источники.
- Интерфейс добавления нового источника по единому интерфейсу адаптера.

**Nice to have**
- Self-host дверь с Docker + JobSpy скраперами + локальным cron.
- Трекер откликов (статусы, история).
- Несколько профилей/языков.
- Экспорт готового CV в PDF (печать из браузера или `pdf-lib`).
- Опциональные уведомления только в self-host (email/Telegram) — без серверов автора.

**Вне scope (осознанно)**
- Любой скрапинг на публичном хостинге.
- Серверное хранение персональных данных.
- Прокси-сервис для LLM-вызовов (ломает принцип BYO/бесплатности).
- Авто-отклик/auto-apply (юридические и этические риски).

## 11. Этапы реализации

1. **Каркас и приватность.** Статический шелл, провайдер-слой LLM, ввод/хранение/удаление ключа, проверка вызова к OpenAI и Anthropic из браузера.
2. **CV → профиль.** Парсинг PDF/DOCX, черновик профиля, редактор, экспорт/импорт.
3. **До-опрос.** Адаптивная генерация вопросов, «полнота профиля», обогащение `hidden_expertise`. (Главный фокус качества.)
4. **Пул вакансий.** `fetch-jobs.mjs` + 2–3 адаптера, нормализация, `jobs.json`. GitHub Action cron + Pages deploy.
5. **Дашборд + матчинг.** Клиентское ранжирование под профиль, фильтры, «почему подходит», «NEW».
6. **Тейлоринг + вилка.** Рерайт под JD, cover letter, зарплатная оценка.
7. **Опенсорс-полировка.** README с GIF, CONTRIBUTING, шаблон адаптера источника, «good first issue».
8. **Self-host (Дверь B).** Docker compose, воркер, JobSpy, локальный cron, отдельный README.

## 12. Метрики успеха

- **Активация:** доля прошедших до-опрос до конца (целевой прокси ценности).
- **Прирост профиля:** насколько мастер-профиль полнее исходного CV (число подтверждённых фактов сверх CV).
- **Качество матчинга:** доля вакансий в топе дашборда, которые пользователь помечает релевантными.
- **Опенсорс-здоровье:** число форков, внешних PR с новыми источниками, звёзд.
- **Стоимость для автора:** должна оставаться ≈ $0 (контроль лимитов free-тиров job-API).

## 13. Риски и открытые вопросы

| Риск | Митигация |
|---|---|
| Free-тиры job-API имеют лимиты/ToS | Брать только разрешающие агрегацию; кэшировать; не плодить запросы; следить за лимитами. |
| Узкое покрытие без LinkedIn/hh-скрапа на хостинге | Сделать сильным self-host (JobSpy); расширять список ATS-фидов; быть честными в README про покрытие. |
| Ключ в браузере «пугает» нетехнарей | Явное объяснение «ключ только у тебя, к нам не уходит»; ссылка-инструкция как получить ключ; кнопка удаления. |
| LLM-вызовы из браузера упрутся в CORS у будущих провайдеров | Слой `providers/`; при отсутствии CORS — задокументировать, что провайдер требует self-host прокси. |
| Anthropic-флаг `dangerous-direct-browser-access` сигналит о риске | Это допустимо для **собственного** ключа пользователя; явно описать в доках. |
| До-опрос задалбывает и люди бросают | Адаптивность, пачки по 2–3 вопроса, ранняя остановка, прогресс-бар. |
| Стоимость для пользователя (его токены) | Дефолт — экономные модели; показывать примерную стоимость; кэшировать ответы матчинга. |

**Открытые вопросы:**
1. Лицензия: MIT (максимум распространения) vs AGPL (защита от закрытых форков-SaaS)? Для «классного опенсорса с защитой» — скорее AGPL.
2. Название и домен (`*.github.io` или свой домен).
3. Стартовый набор источников вакансий под именно её гео (Remote/Belgrade/EU) — какие 3 брать первыми.
4. Дефолтная модель для экономии токенов пользователя.

### Приложение A — мини-гайд «GitHub Pages с нуля»
1. Создать публичный репозиторий, положить `index.html` в корень.
2. Settings → Pages → Source: ветка `main`, папка `/root` → Save.
3. Через ~минуту сайт доступен по `https://<login>.github.io/<repo>`.
4. Любой `git push` в `main` обновляет сайт.
5. Динамика (вакансии) приходит не от Pages, а от GitHub Actions, который коммитит `data/jobs.json`; Pages просто отдаёт обновлённый файл.
